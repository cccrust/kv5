## 功能規劃

### 1. Pub/Sub（發布/訂閱）

Redis Pub/Sub 是即時訊息傳遞的核心機制，適用於即時通知、聊天室、事件廣播等場景。

#### 1.1 核心概念

- **Channel（頻道）**: 訂閱者接收訊息的管道，類似廣播電台頻道
- **Pattern（模式）**: 可使用 glob 風格的模式匹配多個頻道（例：`news.*` 匹配 `news.sports`、`news.tech` 等）
- **Message（訊息）**: 發布者發送的實際內容

#### 1.2 新增命令

| 命令 | 說明 | RESP 格式 |
|------|------|-----------|
| `PUBLISH <channel> <message>` | 向指定頻道發布訊息，返回訂閱者數量 | Integer |
| `SUBSCRIBE <channel> [channel ...]` | 訂閱一個或多個頻道，進入訂閱模式 | Array |
| `UNSUBSCRIBE [channel ...]` | 取消訂閱，無參數時取消全部 | Array |
| `PSUBSCRIBE <pattern> [pattern ...]` | 訂閱模式匹配的頻道 | Array |
| `PUNSUBSCRIBE [pattern ...]` | 取消模式訂閱 | Array |
| `PUBSUB CHANNELS [pattern]` | 列出活躍頻道（可選 pattern 過濾） | Array |
| `PUBSUB NUMSUB <channel> [channel ...]` | 查詢頻道訂閱數 | Array |
| `PUBSUB NUMPAT` | 查詢模式訂閱數 | Integer |

#### 1.3 行為規格

1. **PUBLISH**: 同步發布，無論是否有訂閱者都返回成功
2. **SUBSCRIBE/PSUBSCRIBE**: 用戶端進入「訂閱模式」，此模式下只回應訊息推送，不接受其他命令（除了 UNSUBSCRIBE/PUNSUBSCRIBE）
3. **訊息格式**: 訂閱模式下推送格式為三元素 Array: `["message", channel, message]` 或 `["pmessage", pattern, channel, message]`
4. **PUNSUBSCRIBE/UNSUBSCRIBE**: 在訂閱模式外調用會返回空回應
5. **饋線（Fire-and-forget）**: Pub/Sub 訊息不持久化，離線訂閱者無法接收

#### 1.4 架構設計

```
Store
├── data: DashMap<String, Entry>      # 現有鍵值儲存
├── channels: DashMap<String, Vec<Channel>>  # 頻道→訂閱者映射
└── patterns: Vec<PatternSubscription>       # 全域模式訂閱列表

struct Channel {
    subscribers: Arc<DashMap<String, Sender<PubSubMessage>>>,
}

struct PatternSubscription {
    pattern: String,
    sender: Sender<PubSubMessage>,
}
```

#### 1.5 實作重點

1. **非阻塞發布**: PUBLISH 必須是非阻塞的，即使有大量訂閱者也要快速返回
2. **訂閱者追蹤**: 每個 Client 連線維護自己的 `Receiver<PubSubMessage>`
3. **訊息傳遞**: 使用 tokio channel (mpsc)，每個訂閱者一個 sender
4. **模式匹配**: 使用現有的 glob_match 函數
5. **連線管理**: client 斷線時自動清理其訂閱記錄

### 2. 交易支援（Transaction）

Redis 交易確保一批命令的原子性執行。

#### 2.1 新增命令

| 命令 | 說明 |
|------|------|
| `MULTI` | 開始交易，後續命令加入佇列 |
| `EXEC` | 執行佇列中的所有命令 |
| `DISCARD` | 清除佇列，結束交易 |
| `WATCH <key> [key ...]` | 監控鍵，樂觀鎖 |
| `UNWATCH` | 取消所有監控 |

#### 2.2 行為規格

1. **MULTI** 後輸入的命令不立即執行，加入佇列
2. **EXEC** 執行佇列所有命令，若被 WATCH 的鍵有變化則全部失敗（回傳空）
3. **DISCARD** 清空佇列並退出交易模式
4. **WATCH** 在 MULTI 前呼叫，樂觀鎖機制
5. 交易中的命令失敗不影響其他命令（無 atomic rollback）

#### 2.3 實作重點

1. **Client 狀態**: 每個 client 需要追蹤是否在 MULTI 模式
2. **WATCH 機制**: 記錄 client 監控的鍵，EXEC 前檢查是否被修改
3. **佇列**: 用 Vec 儲存交易中的命令

## 風險評估

1. **Pub/Sub 阻塞**: PUBLISH 不應阻塞，若訂閱者處理慢，應快速返回
2. **交易效能**: WATCH 的樂觀鎖在高競爭場景可能有效能問題
3. **記憶體**: SLOWLOG 需要控制記憶體使用
