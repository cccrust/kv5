# mini-redis 學習筆記

## 版本資訊

- **版本**: v0.3
- **來源**: [tokio-rs/mini-redis](https://github.com/tokio-rs/mini-redis)
- **目的**: 學習 Tokio 最佳實踐，改進 kv5 架構

## 專案概述

mini-redis 是一個教學專案，示範如何使用 Tokio 建構完整的 Redis client/server。雖然功能不完整，但涵蓋了許多重要的非同步 Rust 模式。

## 核心學習

### 1. 正確選擇 Mutex 類型

**mini-redis 做法** (`src/db.rs`):

```rust
struct Shared {
    // 使用 std::sync::Mutex 而非 tokio::sync::Mutex
    // 因為 critical section 內沒有 .await 點，且 critical section 很小
    state: Mutex<State>,
    background_task: Notify,
}
```

**何時用 std::Mutex**:
- Critical section 內沒有 `.await` 點
- Critical section 很小（純記憶體操作）
- 應該避免在持有多個鎖的複雜場景中使用

**何時用 tokio::Mutex**:
- 需要在 critical section 內 `.await`
- 需要跨 `.await` 持有鎖

**對 kv5 的啟示**:
kv5 的 Store 目前使用 `tokio::sync::RwLock`。如果我們確認大部分操作不需要跨 await 持有鎖，可以考慮改用 `std::sync::Mutex` 配合 `spawn_blocking`。

### 2. DbDropGuard - 有序清理模式

**mini-redis 做法** (`src/db.rs`):

```rust
pub(crate) struct DbDropGuard {
    db: Db,
}

impl Drop for DbDropGuard {
    fn drop(&mut self) {
        self.db.shutdown_purge_task();
    }
}
```

**模式說明**:
- 當 `DbDropGuard` 被 drop 時，自動觸發清理邏輯
- 確保背景任務優雅關閉
- 適用於需要確保資源釋放的場景

**對 kv5 的啟示**:
kv5 的 `Store::start_expiry_task` 目前依賴 `Arc<Store>` 的 drop 順序。可以考慮包裝成 `StoreGuard` 來確保清理邏輯一定執行。

### 3. Pub/Sub 廣播模式

**mini-redis 做法** (`src/db.rs`):

```rust
pub(crate) fn subscribe(&self, key: String) -> broadcast::Receiver<Bytes> {
    let mut state = self.shared.state.lock().unwrap();

    match state.pub_sub.entry(key) {
        Entry::Occupied(e) => e.get().subscribe(),
        Entry::Vacant(e) => {
            // 每個頻道有自己的 broadcast channel，容量 1024
            let (tx, rx) = broadcast::channel(1024);
            e.insert(tx);
            rx
        }
    }
}
```

**優點**:
- 每個頻道獨立的廣播管道
- 容量 1024 防止慢消費者阻塞系統
- 簡潔的 API：`subscribe()` 返回 receiver

**對 kv5 的啟示**:
kv5 目前使用 DashMap 儲存订阅者。可以借鑒這種模式，改用 `broadcast::channel` 實現更乾淨的 pub/sub 架構。

### 4. 連線限制模式

**mini-redis 做法** (`src/server.rs`):

```rust
const MAX_CONNECTIONS: usize = 250;

let limit_connections = Arc::new(Semaphore::new(MAX_CONNECTIONS));

// 取得 permit
let permit = self.limit_connections.clone().acquire_owned().await.unwrap();

// spawn task 時傳入 permit，任務完成後 drop(permit) 歸還
tokio::spawn(async move {
    // 處理連線
    drop(permit); // 歸還 permit
});
```

**優點**:
- 使用 Semaphore 控制並發連線數
- 連線數已滿時自動等待，不丟棄連線
- permit 自動歸還（無論任務成功與否）

**對 kv5 的啟示**:
kv5 目前沒有連線限制。可以加入 `MAX_CONNECTIONS` 限制，防止資源耗盡。

### 5. 優雅的關閉模式

**mini-redis 做法**:

1. **Listener 結構** (`src/server.rs`):
```rust
struct Listener {
    notify_shutdown: broadcast::Sender<()>,
    shutdown_complete_tx: mpsc::Sender<()>,
    // ...
}
```

2. **Shutdown 追蹤** (`src/shutdown.rs`):
```rust
pub(crate) struct Shutdown {
    is_shutdown: bool,
    notify: broadcast::Receiver<()>,
}

impl Shutdown {
    pub(crate) async fn recv(&mut self) {
        if self.is_shutdown {
            return;
        }
        let _ = self.notify.recv().await;
        self.is_shutdown = true;
    }
}
```

3. **Handler 中的優雅關閉**:
```rust
while !self.shutdown.is_shutdown() {
    let maybe_frame = tokio::select! {
        res = self.connection.read_frame() => res?,
        _ = self.shutdown.recv() => return Ok(()),
    };
    // 處理命令...
}
```

**模式說明**:
- 使用 `broadcast` 向所有連線發送關閉信號
- 使用 `mpsc` 追蹤所有連線何時完成
- 每個 Handler 有獨立的 `Shutdown` 實例

**對 kv5 的啟示**:
kv5 目前沒有完整的優雅關閉機制。可以引入類似的模式，確保伺服器關閉時所有連線都能完成正在處理的工作。

### 6. Connection Frame 處理

**mini-redis 做法** (`src/connection.rs`):

```rust
pub struct Connection {
    stream: BufWriter<TcpStream>,  // 緩衝寫入
    buffer: BytesMut,              // 緩衝讀取
}

impl Connection {
    pub async fn read_frame(&mut self) -> crate::Result<Option<Frame>> {
        loop {
            if let Some(frame) = self.parse_frame()? {
                return Ok(Some(frame));
            }

            if 0 == self.stream.read_buf(&mut self.buffer).await? {
                if self.buffer.is_empty() {
                    return Ok(None);
                } else {
                    return Err("connection reset by peer".into());
                }
            }
        }
    }
}
```

**優點**:
- `BufWriter` 減少 syscall 次數
- `BytesMut` 高效的緩衝區管理
- `check()` + `parse()` 兩階段解析，避免不必要的 allocation

**對 kv5 的啟示**:
kv5 的 RESP 解析可以借鑒這種兩階段方式，優化大型 frame 的處理效率。

### 7. TTL 到期處理

**mini-redis 做法** (`src/db.rs`):

```rust
struct State {
    entries: HashMap<String, Entry>,
    expirations: BTreeSet<(Instant, String)>, // 按過期時間排序
}

async fn purge_expired_tasks(shared: Arc<Shared>) {
    while !shared.is_shutdown() {
        if let Some(when) = shared.purge_expired_keys() {
            tokio::select! {
                _ = time::sleep_until(when) => {}
                _ = shared.background_task.notified() => {}
            }
        } else {
            shared.background_task.notified().await;
        }
    }
}
```

**優點**:
- `BTreeSet` 按過期時間排序，快速找到下一個要過期的 key
- `Notify` 機制在有新過期時間時立即喚醒
- 使用 `tokio::time::sleep_until` 精確控制喚醒時間

**對 kv5 的啟示**:
kv5 目前每秒檢查所有 key 的過期時間。這個模式可以大幅減少無效的遍歷。

### 8. 錯誤處理模式

**mini-redis 做法**:

```rust
pub enum Error {
    Incomplete,  // 框架不完整，需要更多數據
    Other(Box<dyn std::error::Error + Send + Sync>),
}

impl From<String> for Error { ... }
impl From<&str> for Error { ... }
```

**對 kv5 的啟示**:
kv5 應該為每個底層模組（如 RESP、Store）定義具體的錯誤類型，然後統一轉換為高層錯誤。

### 優先級建議

1. **高優先級**:
   - 實現 `Connection` frame 處理（改善效能）
   - 實現 `DbDropGuard` 包裝 Store
   - 添加連線限制（Semaphore）

2. **中優先級**:
   - 重構 Pub/Sub 使用 `broadcast::channel`
   - 實現 `Shutdown` 追蹤
   - 實現 BTreeSet 優化的 TTL 處理

3. **低優先級**:
   - 客戶端 library
   - OpenTelemetry 整合

## 總結

mini-redis 展示了许多實用的 Tokio 模式：

1. **std::Mutex vs tokio::Mutex** - 根據是否有 `.await` 選擇
2. **DbDropGuard** - 確保背景任務清理
3. **broadcast channel** - Pub/Sub 和優雅關閉
4. **Semaphore** - 連線限制
5. **BufWriter/BytesMut** - 高效網路 I/O
6. **BTreeSet + Notify** - 高效的 TTL 管理
7. **兩階段 frame 解析** - 避免不必要的 allocation

這些模式可以直接應用到 kv5 的改進中，特別是效能優化和正確的資源管理。
